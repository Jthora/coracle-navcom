# 06-01: Internationalization (i18n)

> Extract all user-facing strings into a key-value system for multi-language support.

**Priority**: LOW (no immediate need) → becomes HIGH when non-English operators adopt NavCom  
**Effort**: HIGH (touches every component with user-facing text)  
**Depends on**: 01-* (two-mode architecture — new components should use i18n from the start)  
**Source**: [navcom-future-risks.md](../../navcom-future-risks.md) §6

---

## Problem

All user-facing strings are hardcoded in Svelte components and TypeScript files. Examples:

- `"Send a message"` in compose bars
- `"You're offline"` in toast notifications
- `"Check-In"` / `"Alert"` in message type buttons
- Button labels, error messages, navigation items

Retrofitting i18n is exponentially harder the more components exist. Every new feature adds more hardcoded strings.

---

## Solution

### Library Choice

**Recommended**: `svelte-i18n` (lightweight, Svelte-native, format-message based)

Alternative: `typesafe-i18n` (compile-time type safety, zero-runtime, but more complex setup)

### String Extraction

```svelte
<!-- Before -->
<button>Send a message</button>

<!-- After -->
<script>
  import {t} from "svelte-i18n"
</script>
<button>{$t("compose.send")}</button>
```

### Key File Structure

```
src/
  locales/
    en.json        # English (source of truth)
    es.json        # Spanish (example)
    index.ts       # i18n initialization
```

```json
// en.json
{
  "compose.send": "Send a message",
  "compose.placeholder": "Type a message...",
  "status.offline": "You're offline",
  "mode.comms": "Chat",
  "mode.map": "Map",
  "mode.ops": "Ops",
  "message.checkin": "Check-In",
  "message.alert": "Alert",
  "message.sitrep": "SITREP",
  "message.spotrep": "SPOTREP"
}
```

### Key Naming Convention

- Dot-separated hierarchy: `{domain}.{element}.{qualifier}`
- Examples: `nav.mode.comms`, `channel.header.members`, `crypto.tier.encrypted`
- Pluralization: `"channel.members": "{count, plural, one {# member} other {# members}}"`

---

## Implementation Strategy

### Phase A: Infrastructure

1. Install `svelte-i18n`
2. Create `src/locales/en.json` with all strings
3. Initialize i18n in app entry point with English as default
4. Add locale detection (browser language, user preference)

### Phase B: Progressive Extraction

Don't try to extract all strings at once. Instead:
1. New components use `$t()` from day one
2. When touching an existing component for any reason, extract its strings
3. Dedicated pass for high-visibility strings (nav, modals, errors) in one batch

### Phase C: Translation Workflow

1. Export `en.json` as the base
2. Translators produce `es.json`, `fr.json`, etc.
3. CI check: verify all keys in `en.json` exist in other locale files

---

## Operator Customization

NavCom operators may want to customize terminology (e.g., "Team" instead of "Channel", "Base" instead of "Group"). The i18n system naturally supports this:

```json
// operator-overrides.json
{
  "channel.label": "Team",
  "group.label": "Base"
}
```

Load operator overrides after locale files, merging on top.
