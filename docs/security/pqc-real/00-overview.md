# PQC Real Crypto Implementation — Program Overview

Status: Active
Owner: Core Team (Human-Verified)
Created: 2026-03-20
Goal: Replace placeholder crypto with real encryption so Earth Alliance operatives can securely communicate via decentralized PQC group chats.

## Problem Statement

The existing PQC subsystem under `docs/security/pqc/` and `src/engine/pqc/` was generated by an AI coding agent in a single session (~24,000 lines across 266 files). The architecture, type system, state machines, negotiation, policy, epoch management, and test harness are structurally sound. However, **every "encrypt" operation is actually base64 encoding** — no cryptographic primitives are called anywhere in the transport path.

Specifically:
- `dm-envelope.ts`: `ct: encodeToBase64(plaintext)` — plaintext transmitted in the clear
- `group-epoch-content.ts`: `ct: encodeUtf8Base64(plaintext)` — same issue
- `group-epoch-decrypt.ts`: "decryption" is `decodeUtf8Base64(ct)` — just base64 decode
- `group-epoch-state.ts`: 186-line hand-rolled SHA-256 instead of Web Crypto
- Zero crypto library dependencies in `package.json`
- Zero `crypto.subtle` calls in the transport path
- Deterministic all-zero nonces where random nonces are required

Real AES-GCM-256 encryption exists only in `group-secure-storage.ts` (local state storage), which serves as a correct reference implementation.

## Target Capability

> **Users can log in and use PQC for authentic, fully secure group chats that are decentralized, so they can safely communicate while working as part of the Earth Alliance.**

This means:
1. Group messages are encrypted with real AEAD (AES-GCM-256) before leaving the client
2. Epoch keys are derived using real cryptographic key exchange (X25519 + ML-KEM-768 hybrid)
3. Only current group members can decrypt messages
4. Removed members cannot decrypt future messages (forward secrecy via epoch rotation)
5. Messages are authenticated (sender binding via associated data)
6. The system works over Nostr relays without requiring relay trust
7. Classical fallback uses real NIP-44 encryption, not base64

## Strategy: Phased Crypto Injection

We do NOT rewrite the architecture. The existing scaffolding is correct. We inject real crypto into the defined slots.

### Phase 1 — Crypto Foundation (this doc suite)
Install real crypto libraries, create a `CryptoProvider` abstraction, replace hand-rolled SHA-256 with Web Crypto.

### Phase 2 — Group Epoch AEAD
Wire real AES-GCM-256 encryption into `group-epoch-content.ts` encode/decode using epoch-derived symmetric keys.

### Phase 3 — Hybrid KEM for Key Exchange
Wire ML-KEM-768 + X25519 hybrid key encapsulation into the epoch key distribution path for group key exchange.

### Phase 4 — DM Envelope Encryption
Wire the hybrid KEM + AEAD into `dm-envelope.ts` for encrypted direct messages.

### Phase 5 — Integration & Hardening
Enable the secure pilot flag, run real integration tests with actual encrypted traffic, validate on relays.

## Key Design Decisions

### Why not rewrite from scratch?
The existing architecture (transport adapter pattern, epoch state machine, capability negotiation, feature flags, type contracts) is structurally correct. Rewriting forfeits thousands of lines of well-tested logic. We inject crypto into 6-8 functions.

### Why AES-GCM-256 for symmetric AEAD?
Already proven in `group-secure-storage.ts` using Web Crypto API. Browser-native, hardware-accelerated, well-audited. No additional dependencies needed for symmetric encryption.

### Why ML-KEM-768 for PQC key exchange?
NIST FIPS 203 standardized. The `@noble/post-quantum` library from Paul Miller (same author as `@noble/curves` which is already a transitive dependency) provides a pure-JS implementation. 768 security level balances key size (~1,184 bytes public key) against relay payload constraints.

### Why X25519 + ML-KEM hybrid?
Defense in depth. If either primitive is broken, the other protects. X25519 is battle-tested over a decade. ML-KEM is NIST-standardized but new. Hybrid mode combines both shared secrets via HKDF.

### Why not use NIP-44 for group encryption?
NIP-44 is pairwise (sender↔recipient). Group messages need a single ciphertext decryptable by all current members, which requires a shared symmetric epoch key. NIP-44 can be used for key distribution events (wrapping the epoch key to each member).

## Dependency Plan

| Package | Version | Purpose | Size Impact |
|---------|---------|---------|-------------|
| `@noble/post-quantum` | ^0.2 | ML-KEM-768 (KEM encapsulate/decapsulate) | ~45 KB gzipped |
| `@noble/ciphers` | ^1.0 | ChaCha20-Poly1305 (optional alternative to AES-GCM) | ~15 KB gzipped |
| `@noble/hashes` | ^2.0 | HKDF, HMAC-SHA256 (already transitive dep) | 0 KB (already present) |

Note: `@noble/hashes` is already in the dependency tree via `@noble/curves`. Web Crypto `AES-GCM` needs zero new packages.

## Non-Goals

- Changing the existing envelope format (the `PqcEnvelope` type is fine)
- Changing the NIP-EE kind numbers (443, 444, 445, 10051)
- Rewriting the negotiation, policy, or adaptive control systems
- Building a custom crypto library
- Supporting non-browser runtimes initially (Node.js support comes later)
